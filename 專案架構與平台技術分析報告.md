## 1. 執行摘要

這個專案本質上是「排程式資料管線＋純靜態網站」，不是傳統的常駐後端網站。

每天由 GitHub Actions 啟動 Python 爬蟲，從 acgsecrets.hk 取得動畫季度資料，將封面搬到 Cloudinary，產生季度 JSON，再由 Cloudflare Pages 建置並發布靜態頁面。使用者瀏覽網站時，瀏覽器直接讀取 JSON，不會呼叫你自己的後端 API。

整體架構有幾個明顯優點：

- 靜態網站的主機成本低、速度快、攻擊面比常駐後端小。
- 爬蟲與使用者流量分離，使用者不會直接觸發來源網站請求。
- 已有 HTTP retry、timeout、Pydantic 資料模型、Cloudinary 圖片快取、Discord 通知及 Sentry。
- 目前 36 個季度 JSON 都能正常解析，沒有空檔與 error sentinel。
- 2026 夏季已有 75 筆資料，2026 秋季已有 1 筆預告資料，最新資料產生時間為 2026-07-10。

目前最需要優先處理的問題不是畫面功能，而是安全、資料完整性與部署一致性：

1. .env 仍被 Git 追蹤，且歷史中出現過實際憑證，應視為已外洩。
2. 爬蟲例外會被轉成非空的 error list，可能被當作正常動畫資料寫入並自動提交。
3. Cloudinary 自動清理可能刪除仍被其他季度共用的圖片，也可能在雲端刪除失敗時照樣刪除本地 JSON。
4. static/js/main.js 與實際發布用的 dist/static/js/main.js 已明顯分叉；新版修正不一定會被部署。
5. 目前 2,067 筆動畫的 bangumi_id 全部是「未知ID」，代表來源欄位解析已失效。
6. 專案沒有測試、資料 schema gate、lint、依賴安全掃描或部署 smoke test。

綜合判斷：這是一個已能穩定產出內容的實用型專案，但目前仍屬於「功能可用、營運風險偏高」的狀態。先修安全與資料保護，再做架構重構，會比直接增加新功能更有價值。

### 2.2 Cloudflare 的輕量建置路徑

Cloudflare Pages 執行 build.sh 時會：

1. 設定 BUILD_ONLY=true。
2. 只安裝 Jinja2 與 sentry-sdk。
3. 執行 generate_static.py。
4. 跳過爬蟲及 Cloudinary 套件匯入。
5. 掃描既有季度 JSON 並重新產生 dist/index.html。

這個設計能降低 Cloudflare 建置時間，但目前沒有同步 static 目錄到 dist/static，造成原始前端與發布前端分叉。

### 2.3 瀏覽器執行模式

網站載入後由 Alpine.js 管理畫面狀態：

- 讀取 window.SERVER_DATA 中的年份、季度及建置版本。
- 從 data/年份_季節.json 取得動畫資料。
- 以 localStorage 保存年份、季節、星期篩選及捲動位置。
- 在瀏覽器內執行關鍵字與星期篩選。
- 透過 html2canvas 產生分享圖片。
- 使用 Clipboard API 複製文字或圖片。

因此網站沒有登入、session、伺服器 API 或資料庫查詢。

## 4. 核心程式流程

### 4.1 generate_static.py

主要入口是 generate_static_files()，位於第 93–200 行。

執行順序：

1. 讀取 BUILD_ONLY。
2. 以 datetime.now() 計算目前年份與季度。
3. 確保 dist/data 存在。
4. 若完全沒有 JSON，從 2018 冬季開始全量處理到未來一季。
5. 一般每日模式則更新過去四季、當季與未來一季，共六季。
6. 每季呼叫 generate_quarterly_data()。
7. 掃描所有 JSON 檔名，建立年份與季度選項。
8. 用 Jinja2 產生 dist/index.html。

Build-only 模式只掃描現有資料並產生 HTML，不執行網路爬取。

### 4.2 services/anime_service.py

fetch_anime_data() 位於第 395–492 行，流程為：

1. 將季節轉成 01、04、07、10 月。
2. 檢查 Cloudinary credit 使用率是否達 90%。
3. 若超額，最多清理三個最舊季度。
4. 向來源頁面發 GET，timeout 10 秒。
5. 以 div#acgs-anime-list div.CV-search 選取動畫項目。
6. 把每筆 HTML 交給 multiprocessing.Pool。
7. Worker 解析名稱、星期、時間、圖片與簡介。
8. 下載圖片、計算 MD5、命中快取或上傳 Cloudinary。
9. 用 Pydantic Anime 驗證並輸出 dict。
10. 依星期與時間排序。
11. 保存 cloudinary_cache.json。
12. 發 Discord 成功通知。

HTTP retry 目前針對 429、500、502、503、504，最多三次，backoff factor 為 0.5。

### 4.3 圖片保存

圖片先完整下載，再以內容 MD5 形成：

- Cloudinary public ID：anime_covers/內容雜湊
- 本地 cache key：cloudinary_內容雜湊

優點是跨動畫、跨季度可去除相同圖片的重複上傳；缺點是清理時必須做引用檢查，不能直接把某季度出現過的 public ID 全刪除。

### 4.4 前端

static/js/main.js 的重要區段：

- 第 1–28 行：Alpine 狀態。
- 第 31–115 行：初始化與 localStorage 還原。
- 第 131–201 行：季度 JSON 載入及重試。
- 第 203–220 行：關鍵字與星期篩選。
- 第 242–404 行：分享清單與圖片生成。

一般動畫卡片用 x-text 顯示名稱與簡介，具基本 HTML 跳脫效果；分享圖片區則使用動態 HTML 字串，安全性取決於部署到 dist 的版本。

## 6. 平台與環境設定

### 6.1 GitHub Actions

工作流程：.github/workflows/crawler.yml

| 項目 | 目前設定 |
|---|---|
| 排程 | 每天 UTC 17:00，即台灣時間 01:00 |
| 手動執行 | workflow_dispatch |
| Push 觸發 | 只監看 templates/** 與 static/** |
| Runner | ubuntu-24.04 |
| Python | 3.11 |
| 時區 | TZ=Asia/Taipei |
| pip cache | 開啟 |
| 主要指令 | python generate_static.py |
| 自動提交 | dist/data/ 與 cloudinary_cache.json |
| 併發 | 同一 workflow＋ref 排隊；不取消進行中工作 |

GitHub Secrets 需求：

- CLOUDINARY_CLOUD_NAME
- CLOUDINARY_API_KEY
- CLOUDINARY_API_SECRET
- DISCORD_WEBHOOK_URL

需要注意：

- Push 沒有限定 main，因此其他分支修改 templates 或 static 也可能操作同一個 Cloudinary 帳號。
- concurrency group 包含 ref，只能防止同一分支互撞；不同分支仍可能同時操作共享資源。
- 沒有明確 permissions: contents: write，推送能力依賴儲存庫外部設定。
- fetch-depth 設為 1，但旁邊註解稱要完整歷史，兩者矛盾。
- 修改 Python、requirements、workflow、config 或 build.sh 不會立即觸發此工作流程。
- action 使用 checkout@v4 與 setup-python@v5 標籤，沒有固定 commit SHA。
- 沒有 PR 測試、資料驗證、lint、套件掃描或部署 smoke test。

## 7. 資料模型與目前資料狀態

### 7.1 JSON schema

每個季度檔的概念結構：

~~~json
{
  "anime_list": [
    {
      "bangumi_id": "來源 ID 或未知ID",
      "anime_name": "動畫名稱",

~~~

### 7.2 2026-07-10 最新盤點

| 指標 | 數值 | 解讀 |
|---|---:|---|
| 季度 JSON | 36 | 2018–2026 |
| JSON 總大小 | 約 1.32 MB | 靜態發布仍很輕量 |
| 動畫紀錄 | 2,067 | 全部能被 JSON parser 讀取 |
| 空季度檔 | 0 | 2026 夏 75 筆、秋 1 筆 |
| error sentinel | 0 | 現存資料未被 error dict 污染 |
| Cloudinary 圖片 | 2,067 | 現存紀錄全部使用 Cloudinary |
| bangumi_id=未知ID | 2,067 | 100%，ID 解析已失效 |
| 暫無簡介 | 939 | 約 45.4% |
| 無首播日期／時間 | 各 5 | 少量資料不完整 |
| Cloudinary cache | 1,942 筆，約 336 KB | 少於紀錄數是因內容去重 |

所有 generated_at 都是沒有時區 offset 的 naive timestamp。網站會用瀏覽器所在地解讀時間，非台灣使用者看到的更新時間可能有語意偏差。

### 7.3 Git 同時扮演的角色

目前 Git 不只存原始碼，也保存：

1. 36 份季度 JSON。
2. Cloudinary 圖片索引快取。
3. 部分建置產物。
4. Cloudflare Pages 部署來源。

好處是簡單、可回溯；代價是自動資料更新、程式版本與部署歷史緊密耦合，repository 歷史會持續成長。

## 8. 安全、資料完整性與維運風險

### P0：必須立即處理

#### 8.1 .env 已被 Git 追蹤

證據：

- .gitignore 第 4 行雖然包含 .env。
- git ls-files 仍列出 .env。
- Git 歷史至少有三個含 .env 的提交。
- 現有七個變數都有非空、非 placeholder 的值。
- 歷史出現過 Cloudinary 憑證、兩組帳密與 LINE_TOKEN。

結論：

只要這個 repository 曾推送到 GitHub，就應把這些值視為已外洩；即使 repository 是 private，也不能只靠刪除目前檔案來解除風險，因為舊 commit 仍保留內容。

建議：

1. 先輪替 Cloudinary API secret、兩組帳密與仍可能有效的 LINE token。
2. 確認 GitHub Secrets、Cloudinary audit log 及相關帳號登入紀錄。
3. 將 .env 從 Git index 移除。
4. 建立只含變數名稱的 .env.example。
5. 視 repository 分享範圍決定是否重寫 Git 歷史。

### P1：高風險

#### 8.2 系統錯誤可能被寫成正常動畫資料

services/anime_service.py 第 489–492 行會把任何例外轉成：

~~~text
[{"error": "系統錯誤: ..."}]
~~~

同一函式在無效季節與 Cloudinary 清理失敗時也會回傳類似的非空清單。

generate_static.py 第 66–88 行只把空 list 視為失敗；error list 是非空，因此可能被寫入正式季度 JSON。GitHub Actions 隨後會自動提交，形成「工作流程成功，但資料已污染」。

而 generate_static.py 第 47–64 行想捕捉的網路例外，通常已先被服務層吞掉，實際上很難生效。

建議：

- 服務層只回傳合法動畫清單。
- 系統錯誤一律 raise 具名例外。
- 寫檔前重新用 schema 驗證整份資料。
- 只要出現 error key、筆數異常或欄位比例超標，就以非零狀態停止 Action。
- 寫入採暫存檔＋原子替換，避免中途破壞舊資料。

#### 8.3 Cloudinary 清理可能破壞其他季度

圖片是以內容 hash 跨季度共用，但 cleanup_specific_season() 只依被刪季度建立 public ID 清單，沒有檢查其他季度是否仍引用同一張圖片。

最新資料的實測：

- 有 161 個圖片 URL 被兩筆以上紀錄共用。
- 共有 340 筆紀錄使用共享 URL。
- 最多同一 URL 被三個季度引用。
- 最舊的 2018 冬季有 59 張圖片，其中 14 張也被其他季度引用。

如果配額達 90% 並自動清理 2018 冬季，這 14 張共用圖片可能一起被 Cloudinary 刪除，造成其他季度的封面失效。

此外，Cloudinary 批量刪除失敗後，程式仍會繼續刪除本地季度 JSON；workflow 又會以 git add --all dist/data 提交該刪除。

建議：

- 動畫文字 JSON 不應因圖片額度不足而被刪除。
- 清理前建立全資料集引用計數，只刪除引用數為零的圖片。
- 加入 dry-run、備份、兩階段確認與刪除結果核對。
- 雲端刪除未完全成功時，禁止刪本地 JSON 或 cache。
- 清理策略應獨立於每日爬蟲，不應在抓資料流程中自動做破壞性操作。

#### 8.4 原始 JS 與發布 JS 已分叉

實際比對：

- static/js/main.js 最後程式提交：2026-04-07。
- dist/static/js/main.js 最後程式提交：2025-12-08。
- 兩者差異達 208 行。
- static/css 與 dist/static/css 目前相同。

新版 source JS 已加入：

- 網路重試。
- 動畫名稱 HTML 跳脫。
- 剪貼簿失敗時自動下載。
- 更完整的資源釋放。

dist 版仍是舊邏輯，並在分享圖片功能把外部來源的名稱與圖片 URL 直接插入 insertAdjacentHTML，形成 stored-XSS 攻擊面。

build.sh 只執行 generate_static.py，而 generate_static.py 只產生 dist/index.html，沒有同步 static 到 dist/static。因此修改 source JS 即使觸發 workflow，也不保證進入發布目錄。

建議：

- static 只保留一份來源。
- 每次 build 明確複製或打包到 dist/static。
- CI 比較 source／dist hash，分叉就失敗。
- 不追蹤可重建的 dist 資產，或明確規定建置產物更新流程。
- 分享圖片 DOM 改用 createElement 與 textContent，不用外部資料拼 innerHTML。

### P2：中風險

#### 8.5 bangumi_id 解析完全失效

services/anime_service.py 第 270 行讀取 acgs-bangumi-data-id，但最新 2,067 筆全部是「未知ID」。

影響：

- 無法用穩定 ID 做去重與資料更新。
- 前端只能用 anime_name 當 key。
- 名稱變更時無法辨識同一作品。
- 無法建立可靠的增量比較。

目前同一季度沒有重複名稱，所以畫面尚未實際碰撞，但資料契約已失去一個重要欄位。

#### 8.6 沒有資料品質門檻

單筆 Worker 解析失敗時會回傳 None 並被直接略過。如果來源原有 70 筆、這次只成功 20 筆，20 筆仍可能覆寫舊檔。

建議在寫檔前比較：

- 舊筆數與新筆數差異。
- 解析失敗率。
- 未知 ID、無日期、無圖片、無名稱比例。
- 重複 key。
- 來源 DOM selector 命中率。

#### 8.7 Git ignore 與建置產物管理不正確

.gitignore 第 21–23 行把說明文字寫成行尾註解；Git ignore 不把它視為註解，實測 dist 規則沒有生效。

目前仍被追蹤：

- .env
- 多個 __pycache__／pyc
- dist/index.html
- dist/static

這會增加安全風險並放大 source／artifact 漂移。

#### 8.8 依賴與供應鏈不可完全重現

- Python 依賴沒有完整 lock 與 hash。
- lxml、Jinja2 未固定。
- Cloudflare build 另裝未固定版本。
- 前端 CDN 沒有 SRI。
- SweetAlert2 只鎖大版本。
- 沒有 CSP。
- Actions 沒固定 commit SHA。
- 沒有依賴漏洞掃描。
- workflow 把 Cloudinary secret 與 Discord webhook 設成 job-wide env，checkout、第三方 action 與依賴安裝步驟都處在同一秘密環境；應只在真正執行爬蟲的 step 注入。

#### 8.9 Sentry 設定與隱私

前端 Sentry DSN 寫在公開 HTML。瀏覽器 DSN 通常不是 API secret，但應限制允許來源並避免濫用。

目前設定：

- tracesSampleRate：100%
- replaysSessionSampleRate：10%
- replaysOnErrorSampleRate：100%

應確認：

- 流量與費用。
- Replay 是否捕捉使用者輸入或頁面內容。
- 隱私告知與資料留存。
- 正式／預覽環境是否分流。

Python Sentry 需要 SENTRY_DSN 才啟用，但 workflow 沒有設定，因此排程爬蟲端預設沒有 Sentry。

#### 8.10 爬蟲禮儀與來源耦合

來源 selector 非常具體，網站 DOM 改版即可能失效。程式沒有明確 User-Agent、季度間延遲或全域 request rate limit；Worker 數量使用 os.cpu_count()。

建議：

- 檢查來源網站 robots.txt 與使用條款。
- 使用可辨識 User-Agent。
- 限制平行數、增加 jitter 與季度間延遲。
- 對 429 尊重 Retry-After。
- 保存 HTML fixture 供離線測試。

#### 8.11 外部圖片下載缺少安全邊界

worker 直接採用來源 HTML 中的圖片 URL。upload_to_cloudinary() 會允許 redirect、把完整 response 載入記憶體，再上傳到 Cloudinary；目前沒有：

- URL scheme 與 hostname allowlist。
- 每次 redirect 的重新驗證。
- loopback、private、link-local IP 封鎖。
- Content-Type 驗證。
- 最大檔案大小。
- 串流下載上限。
- 真正圖片內容驗證。

如果來源頁面遭入侵或回傳惡意圖片 URL，GitHub runner 可能被利用去存取不應接觸的位址，或下載超大內容耗盡記憶體。

建議只允許預期的 HTTPS 圖片主機；每一跳都重新解析並拒絕內網位址；使用串流、最大 bytes、Content-Type 與影像解碼驗證後才上傳。圖片上傳失敗時，也不應把未驗證的原始 URL直接保存進正式 JSON。
