# 安全維運與災難復原手冊

這份手冊包含會影響外部平台、憑證或資料的操作。每次只做一個主題，保留操作紀錄，遇到不確定就先停下來。

## 1. 緊急處理順序

若秘密曾進入 Git，正確順序固定如下：

1. 暫停 GitHub 每日爬蟲，確認沒有 job 正在執行。
2. 暫停任何 Cloudinary retention／cleanup。
3. 到秘密原本所屬的平台撤銷或輪替。
4. 更新 GitHub Environments 與本機密碼管理器。
5. 用修補後的程式執行一次人工驗證。
6. 確認新憑證正常後停用舊憑證。
7. 最後才另開 Git 歷史清理維護窗口。

刪除 `.env` 或改 `.gitignore` 只能防止再次提交，不能讓已公開的秘密失效。

## 2. 立即暫停自動工作

GitHub：

1. 若目前仍是舊版 crawler workflow，先進入儲存庫 **Actions**，選取舊 workflow 並從右上角選單按 **Disable workflow**。舊版沒有讀取 `CRAWLER_SCHEDULE_ENABLED`，所以只改變數不能阻止它。
2. 取消目前所有 Running 或 Pending crawler jobs；Disable workflow 與變數都不會中止已經開始的工作。
3. 確認沒有 crawler 正在上傳 Cloudinary、建立 automation PR 或等待 auto-merge。
4. 進入 **Settings → Secrets and variables → Actions → Variables**，將 repository variable `CRAWLER_SCHEDULE_ENABLED` 設為 `false`，或先刪除／保持未設定。
5. 新版 workflow 合併到 `main` 後，這個變數才會讓下一次排程安全跳過；新版仍可保留人工 **Run workflow** 驗證能力。人工驗證前不要重新開排程。

Cloudinary：

- 不要執行 `cloudinary_cleaner.py --execute`。
- 不要在 Cloudinary Console 使用 Bulk Delete。
- 一般 `generate_static.py` 已不含自動刪季度／刪圖邏輯，但在完成部署驗收前仍保持排程暫停。

## 3. Cloudinary 無停機輪替

此專案需要 API key 與 API secret 才能上傳、查詢額度及建立 retention plan。輪替時先讓新舊兩組短暫並存，避免網站更新中斷。

### 3.1 輪替前

- [ ] 確認 GitHub workflow 已暫停。
- [ ] 確認 `CRAWLER_SCHEDULE_ENABLED` 未設定或為 `false`。
- [ ] 確認程式不會自動刪除資源。
- [ ] 記錄目前 Cloudinary product environment 名稱；不要把 secret 寫進紀錄。
- [ ] 到 Cloudinary **Settings → Backup** 啟用 automatic backup。
- [ ] 對 existing assets 執行初始備份，等待狀態完成。
- [ ] 在 Assets 建立一張測試圖片，實際演練刪除與 restore，確認帳號方案可復原。

Automatic Backup 預設可能是關閉的，而且備份會計入儲存額度。沒有在刪除前啟用備份，就不能假設資產一定能救回。

### 3.2 產生新 key

1. Cloudinary Console → **Settings → API Keys**。
2. 選 **Generate New Access Key**。
3. 名稱建議 `anime-crawler-prod-YYYY-MM`。
4. 將新 API key 與 secret 直接存入密碼管理器。
5. 若 Console 有 **Role Management**，新 key 預設沒有任何權限。回到
   **Settings → API Keys**，為新 key 指派目前 product environment 的角色；為了與
   舊 key 的既有功能相容，可先使用 product-environment **Master Admin**（不是
   Account Master Admin）。此專案需要 Admin API 的資產／用量讀取、Upload API 上傳，
   retention execute 另需要刪除權限。Enterprise 若可建立 granular custom role，應改用
   分離 key 與最小權限：crawler 只讀用量並上傳、plan 只讀資產、execute 只讀並刪除。
6. 不要停用舊 key。

Cloudinary 2026 Roles and Permissions 對新 key 採零權限預設；只有 key/secret 正確並不
代表它看得到資產。若 dry-run 顯示季度資料仍有大量 `referenced`，卻出現
`inventory=0`，視為輪替失敗並保留舊 key。新版工具會對空 inventory、空頁但仍有
continuation cursor、重複 cursor 或越出 `anime_covers/` 的資源直接失敗。

### 3.3 更新使用位置

GitHub **Settings → Environments**：

1. 在 `crawler-production` 更新 Secrets `CLOUDINARY_API_KEY` 與 `CLOUDINARY_API_SECRET`。
2. 在 `cloudinary-retention-plan` 與 `cloudinary-retention-execute` 更新相同兩項 Secrets。
3. Variable `CLOUDINARY_CLOUD_NAME` 通常不變；若換 product environment，三個 environments 都要更新。
4. 先保留 Repository-level 同名舊 secrets，直到 Environment secrets 完成兩次人工驗證；驗證後到 **Settings → Secrets and variables → Actions → Repository secrets** 刪除同名舊項目，避免日後誤更新。

本機：

- 更新未追蹤的 `.env`。
- 更新密碼管理器。
- 不要把新值放進 Cloudflare Pages；Cloudflare build-only 不需要。

### 3.4 驗證再停用舊 key

1. 部署修補後的 `main`，確定品質 gate 與 Cloudflare build 全綠。
2. 確認新版 workflow 已合併並重新 Enable，保持 `CRAWLER_SCHEDULE_ENABLED=false`，手動執行一次 GitHub 爬蟲；新版的手動 `workflow_dispatch` 不受此排程開關阻擋。
3. 確認能查詢 Cloudinary、命中既有 cache，並在必要時成功上傳一張圖片。
4. 執行 retention dry-run，確認 `inventory` 大於 0；候選數字只供檢查，不執行
   `--execute`。
5. 確認沒有任何 delete log，舊季度圖片仍能開啟。
6. 到 Cloudinary 將舊 key 的 Status 改為 **Disabled**。
7. 再手動執行一次爬蟲或非破壞性查詢，確認只有新 key 仍可使用。
8. 保留輪替日期、key 名稱與驗證結果，不記錄 secret 本身。
9. Environment secrets 已驗證後，刪除 Repository-level 的 Cloudinary 同名 secrets；再次手動執行，確認沒有依賴舊位置。

完成實站驗收並觀察穩定後，才把 repository variable `CRAWLER_SCHEDULE_ENABLED` 改成小寫 `true`。其他值、空白或大寫 `TRUE` 都不會開啟排程。

若新 key 失敗，在舊 key 尚未停用前先回復 GitHub secret；不要反覆建立更多 key。

## 4. 兩組歷史帳密

歷史中的 `USERNAME_1`、`PASSWORD_1`、`USERNAME_2`、`PASSWORD_2` 已不被現行程式使用，`.env.example` 也不應再出現它們。

處理方式：

1. 從自己的密碼管理器或舊部署紀錄辨識每組帳密屬於哪個服務。
2. 在該服務官方網站直接更改密碼。
3. 選擇「登出所有裝置」或撤銷所有現有 session。
4. 撤銷同帳號的 app password、PAT、API token 或 SSH key。
5. 啟用 MFA，保存 recovery codes 到密碼管理器。
6. 若密碼曾重複使用，所有重複的服務都要換；只改其中一個不夠。
7. 刪除 GitHub Repository／Environment 中同名的舊 secrets。
8. 不要為了「保持相容」建立新的 `USERNAME_1` 等變數；程式已不需要。

如果無法辨識服務，仍要把任何重複使用的密碼全面更換，並檢查常用帳號的登入紀錄。Git 歷史清理不能代替改密碼。

## 5. 歷史 LINE token

先判斷 token 類型：

- 若是 LINE Notify token：LINE Notify 已於 2025-03-31 終止，所有 Notify API 已停用。現行專案不需要 replacement，移除 GitHub、本機與歷史即可。
- 若是 LINE Messaging API channel access token：到 LINE Developers Console 撤銷。現行專案沒有使用 LINE，不要補發新 token。
- 若是 LINE Login user access token：使用 LINE revoke endpoint 或解除應用授權。

最後確認：

- GitHub Secrets／Environments 沒有 `LINE_TOKEN`。
- `.env` 與 `.env.example` 沒有 `LINE_TOKEN`。
- Sentry、GitHub Actions log、Discord 訊息沒有留下 token。

## 6. Discord 與 Sentry 安全

### Discord

Webhook URL 任何人拿到都能向頻道發訊息。若曾進 Git、log 或聊天：

1. Discord 頻道 → **Integrations → Webhooks**。
2. 刪除舊 Webhook；只重新命名不會換 URL。
3. 建立專用新 Webhook。
4. 更新 GitHub `crawler-production` 的 `DISCORD_WEBHOOK_URL`。
5. 執行一次人工 workflow，確認成功與失敗訊息不含 Webhook URL、secret、token 或來源 HTML。GitHub Pull Request／Actions run 的可點擊網址是預期的維運資訊。

正式每日排程必須有可用的 Discord webhook；未完成上述驗證前，`CRAWLER_SCHEDULE_ENABLED` 保持 `false`。本機開發可不設定 webhook，不能把這個本機例外套用到正式排程。

### Sentry

- `SENTRY_DSN`：爬蟲端設定，放 GitHub `crawler-production`。
- `SENTRY_BROWSER_DSN`：前端公開 client key，放 Cloudflare Production environment variable；GitHub `crawler-production` 也以同名 Variable 供爬蟲建置驗證使用。
- Browser Allowed Domains 只允許正式網域與實際 Pages 網域，不使用 `*`。
- `_headers` 的 CSP `connect-src` 必須允許 Browser DSN 的區域主機；目前允許 `*.ingest.sentry.io`、`*.ingest.us.sentry.io`、`*.ingest.de.sentry.io`。DSN 主機不在清單時先修 CSP 並測試，不要放寬成 `*`。
- 開啟 Spike Protection，為 client key 設 rate limit。
- Session Replay 維持 0；若要啟用，須另做隱私告知、遮罩、費用與 CSP 評估。
- 不開啟 network request／response body 收集。
- profiles sample rate 維持 0，除非有明確效能調查。

Browser DSN 本來就會出現在 HTML，不需要像密碼一樣藏起來；防護重點是來源限制、速率限制與資料最小化。

## 7. Cloudinary retention：只規劃，不自動刪

### 7.1 安全規則

目前工具遵守：

- 預設 dry-run。
- 不刪除任何季度 JSON。
- 先對全部 `dist/data/*.json` 做完整 schema 驗證，再建立全域引用集合；任一 JSON 錯誤就拒絕規劃或刪除。
- 只有 `anime_covers/` 中「沒有任何 JSON 引用」且至少 30 天的圖片才是候選；`--minimum-age-days` 無法降到 30 以下。
- 受管 public ID 僅允許 `anime_covers/` 加上舊版 32 位 MD5 或新版 64 位 SHA-256 小寫十六進位摘要；任意名稱、巢狀子目錄、大小寫或其他長度都會 fail closed。這是為了相容既有資產，不會擴張到其他 Cloudinary 資料夾。
- manifest 使用 `schema_version: 3`，內含 Cloudinary `cloud_name` 與依 canonical JSON 計算的 `manifest_sha256`；讀取時會做完整欄位、型別、計數、一致性、路徑、重複值、cloud 綁定與 SHA-256 驗證。
- manifest 至少等待 30 天；`--grace-days` 無法降到 30 以下。
- 正式 Execute 會以 GitHub API 的 dry-run `created_at` 當可信時間，要求該 run 已滿 30 天，且 manifest 必須在 run 開始後 30 分鐘內產生；不能只修改 manifest 時間與重算無金鑰 digest 來縮短等待。
- execute 前會再次完整驗證 manifest、重新掃描全部資料與 Cloudinary，最後只刪「舊 manifest 與目前候選的交集」。
- 每次硬性最多 50 張且不超過 Cloudinary inventory 的 2%；命令列只能設得更保守，不能提高上限。
- 在不可逆的 Cloudinary 刪除呼叫**之前**，先移除所有候選的 cache entries 並原子保存。若外部刪除只完成一部分，後續爬蟲不會沿用已刪圖片的舊 URL，而會重新檢查／上傳。
- 任一批次沒有逐項確認成功就停止並回報；JSON 永遠不會被 retention 修改。

額度不足時的正確行為是讓爬蟲失敗、通知維運者並保留舊資料；不能用刪歷史季度救額度。

### 7.2 建立 dry-run manifest

前提：

- Cloudinary automatic backup 與 existing assets 初始備份完成。
- `.env` 有目前有效的三個 Cloudinary 值。
- 已安裝 `requirements.txt`。
- `python manage.py validate-data` 成功。

PowerShell：

```powershell
python cloudinary_cleaner.py `
  --minimum-age-days 30 `
  --manifest-output retention-plan-2026-07-10.json
```

Linux、macOS、Git Bash：

```bash
python cloudinary_cleaner.py \
  --minimum-age-days 30 \
  --manifest-output retention-plan-2026-07-10.json
```

`retention-plan*.json` 已被 `.gitignore` 忽略。它不是 secret，但屬於維運紀錄，請保存到受控資料夾或加密備份，不要提交。

manifest 自己已包含 `manifest_sha256`，工具在 execute 前會重新計算並拒絕任何內容變更。另記錄整個檔案的外部 SHA-256，方便維運者在等待 30 天期間核對檔案：

```powershell
Get-FileHash retention-plan-2026-07-10.json -Algorithm SHA256
```

### 7.3 人工審查 manifest

打開 JSON，確認：

- `schema_version` 是 3，且 `cloud_name` 必須是你這次審查的 Cloudinary product environment。
- `manifest_sha256` 存在；不要手動修改任何欄位或重算它來掩蓋變更。
- `created_at` 有時區。
- `minimum_age_days` 至少 30。
- `inventory_count` 合理，沒有突然變成 0 或少很多。
- `referenced_count` 與目前 cache／資料規模相近。
- `delete_candidates` 只在 `anime_covers/`。
- 候選沒有出現在任何季度 JSON。

抽查 Cloudinary Assets 中至少 10 筆候選；候選少於 10 就全部查。任何數字異常、JSON 驗證失敗、Cloudinary 分頁錯誤或來源不明，都丟棄 manifest，不要執行。

### 7.4 等待 30 天

從 manifest 的 `created_at` 起算至少 30 個完整日曆日。期間：

- 可以正常更新動畫資料。
- 不可修改 manifest 的 `created_at` 來縮短等待。
- 若秘密輪替，只需用新 key 執行；manifest 候選本身仍會在執行前重新交集。
- 若發現候選其實被使用，修正資料後重新產生 manifest，等待期重新開始。

### 7.5 30 天後人工執行

正式刪除只能使用 GitHub 的 **Cloudinary Retention Execute** 手動工作流程；舊的一步式 `--execute` 已永久拒絕，只有受保護 workflow 能依序使用 `--prepare-execution` 與 `--execute-prepared`。這能保證使用 fresh `main`，並以共用 concurrency lock 避免與 crawler 同時操作 Cloudinary。

> 升級到目前的兩階段 retention 流程後，舊版 manifest schema 不可沿用。請在新版 **Cloudinary Retention Dry Run** 重新產生計畫，重新人工審查並重新等待至少 30 天；不要為了沿用舊計畫而改檔或略過等待期。

1. 先把 repository variable `CRAWLER_SCHEDULE_ENABLED` 設為小寫 `false`，取消尚未開始的排程，並確認沒有 Running／Pending crawler，且所有以 `main` 為目標的 Pull Request 都已合併或關閉。只 Disable 排程事件不足以中止已在執行的 job。
2. 再次確認 Cloudinary backup 正常。
3. 從 30 天前的 **Cloudinary Retention Dry Run** 網址取得 run ID；網址 `/actions/runs/123456789` 中的數字就是 run ID。
4. GitHub → **Actions → Cloudinary Retention Execute → Run workflow**，Branch 必須選 `main`。
5. `plan_run_id` 填入原 dry-run run ID；`confirmation` 完整輸入 `DELETE_UNREFERENCED`。
6. `cloudinary-retention-execute` 的 required reviewer 再次核對 manifest、外部 SHA-256、備份、等待天數、沒有任何 open main PR，以及 `CRAWLER_SCHEDULE_ENABLED=false` 後才核准。若 GitHub 對後段 delete job 再次要求 environment approval，仍須重新核對後才批准。
7. workflow 會透過 GitHub API 確認原 run 的固定來源檔是 `.github/workflows/retention-plan.yml`（相容 API 回傳的純路徑或附加 main ref 格式），並另外核對 `head_branch=main`、成功狀態、人工觸發來源與可信 `created_at`；接著才下載該 run 的 artifact、重新 checkout 最新 main 並驗證全部資料，任何條件不符就失敗。
8. 第一階段只產生自我雜湊的 execution receipt 並從本機 cache 移除候選 URL，**此時尚未刪除任何 Cloudinary 圖片**。無秘密的 publisher 會建立 cache-only PR；必須等 `Quality Gate / quality` 全綠並 auto-merge，後段 delete job 才能開始。從 cache PR 合併到 Cloudinary delete 完成前，不要建立或合併其他 main PR。

工具會先重新驗證 manifest 的 schema 與內建 SHA-256，再重新驗證全部季度 JSON、計算引用並重新列出 Cloudinary。若候選超過 50 或 inventory 的 2%，會整次拒絕，不是自動只刪前幾筆。此時請停下來人工調查，不要提高上限、拆批、改程式或用 Cloudinary Bulk Delete 規避保護。

兩階段順序是安全契約：`prepare-cache-invalidation` 重驗 aged manifest、目前引用、Cloudinary inventory 與 50 張／2% 上限，原子產生 cache 與 receipt artifact，但絕不呼叫 delete API；`publish-cache-pr` 不持有 Cloudinary secrets，只能把 `cloudinary_cache.json` 經 PR、strict ruleset 與 Quality Gate 合併；`delete-cloudinary` 再 checkout 該 cache-safe main SHA，核對 receipt 自我雜湊、Cloudinary cloud name、manifest hash、cache byte hash、base-to-merge 唯一差異、遠端 main SHA、無 open PR 與 cache 已無候選 URL，全部成立才第一次呼叫 delete API。App 與 GitHub Actions 都不具 ruleset bypass。

若 prepare、artifact、cache PR、Quality Gate 或合併失敗，workflow 必須紅燈且完全不會刪圖；修正後從 fresh main 重新執行。若 cache PR 已合併但 delete 前置驗證或 environment approval 未完成，cache 只是提早停止重用仍存在的候選圖片，屬於安全偏保守狀態；保持排程 `false` 並從 fresh main 重跑，不要復原舊 URL。若 Cloudinary 回傳部分失敗、timeout 或 runner 中斷，main cache 已預先失效，但整體 workflow 仍必須紅燈；重新做 dry-run 盤點剩餘資產，不要盲目重播舊 receipt。候選為 0 時不建 PR也不刪圖；候選存在但 cache 原本已安全時可不建 no-op PR。任何情況都不可直接推送 `main` 或使用 ruleset bypass。

執行後：

1. 保存 log、manifest hash、實際刪除數與操作者。
2. 執行 `python manage.py validate-data`。
3. 抽查最新與最舊季度圖片。
4. 檢查 `cloudinary_cache.json` 在 Cloudinary delete 前就已移除全部候選 public IDs；即使某個候選最後未刪成功，它也會暫時不在 cache，留待後續爬蟲安全重建，不要手動復原舊 URL。
5. 若本次需要 cache PR，確認它已通過 required check 並 auto-merge；再確認 `delete-cloudinary` 成功且整個 retention workflow 綠燈。只看到 receipt artifact、open PR、已合併 cache PR 或部分成功 delete 都不算完成。
6. 完成上述確認後，才可把 `CRAWLER_SCHEDULE_ENABLED` 恢復為小寫 `true`。
7. 觀察網站、Discord 與 Sentry 24 小時。

### 7.6 GitHub dry-run

儲存庫的 **Cloudinary Retention Dry Run** workflow 只產生計畫：

1. GitHub → **Actions → Cloudinary Retention Dry Run → Run workflow**。
2. Branch 選 `main`，`minimum_age_days` 保持 30 或更高。
3. job 會使用 environment `cloudinary-retention-plan`，先驗證全部 JSON，再查詢 Cloudinary。
4. 成功後從該次 run 的 **Artifacts** 下載 `cloudinary-retention-plan-<run id>`。只在剛觸發、尚未超過可信時間窗時重跑同一次 run；若 environment 核准、排隊或重跑使該 run 已超過 20 分鐘，workflow 會立即紅燈，請建立全新的 dry-run，不要重跑舊 run。
5. artifact 保存 90 天；仍應立即保存 manifest 與 SHA-256 到自己的受控維運位置。
6. 核對 workflow log 只有 DRY RUN，沒有 `--execute`、`DELETE_UNREFERENCED` 或 delete API。
7. 依上面流程人工審查並等待 30 天。這個 **Dry Run** workflow 本身沒有刪除步驟；只有另一個受 reviewer 保護、人工觸發且與 crawler 共用鎖的 **Execute** workflow 能刪除。

任何 schedule、push 或 pull request 都不應自動執行刪除。

## 8. 上線前災難復原準備

每次重大架構更新前完成：

- [ ] GitHub 建立清楚的 last-good tag，例如 `pre-architecture-2026-07`。
- [ ] 所有 secrets 在密碼管理器有最新副本；GitHub secret 無法讀回。
- [ ] Cloudinary automatic backup 已啟用。
- [ ] Existing assets 初始備份成功。
- [ ] 實際演練一張測試 asset 的 restore。
- [ ] 匯出 Cloudinary manifest：至少包含 `asset_id`、`public_id`、`version`、`format`、`secure_url`、`bytes`、`created_at` 與 backup 狀態。
- [ ] 記錄 Cloudflare project name、production branch、build command、output directory、pages.dev URL、自訂網域及 DNS。
- [ ] 記錄 Sentry organization／project、Allowed Domains 與採樣率，不記錄管理 token。
- [ ] 本機 `pytest`、ruff、資料驗證與 build-only 全綠。
- [ ] 確認知道如何在 Cloudflare 回滾。

備份本身也要測試。只有看到「Backup enabled」但從未 restore，不算完成演練。

## 9. 災難復原情境

### 9.1 網站新版壞掉

先恢復服務，再修程式：

1. Cloudflare Pages → **Deployments**。
2. 找到上一個成功的 production deployment。
3. 選單 → **Rollback to this deployment**。
4. 確認正式網域恢復。
5. 從最新 `main` 建立復原分支，對壞 commit 執行 `git revert`，推送該分支並開 Pull Request；不要 `reset --hard`，也不要直接推送 `main`。
6. 等 `Quality Gate / quality` 全綠並依 ruleset 合併，再確認新的 Cloudflare build 成功。

Cloudflare rollback 只改目前對外版本，不會修 Git；兩邊都要處理。

### 9.2 JSON 被錯誤資料覆寫

1. Disable 每日爬蟲 workflow。
2. 找到最近一筆 bot data commit。
3. 從最新 `main` 建立復原分支，使用 `git revert <commit>` 建立反向提交。
4. 執行 `python manage.py validate-data` 與 build-only。
5. 推送復原分支、建立 Pull Request，等待 required check 全綠並合併；不要直接推送 `main` 或使用 ruleset bypass。
6. 修 parser fixture 或品質 gate，確認後才恢復排程。

資料寫入採原子替換，因此中途失敗不應留下半個 JSON；Git last-good 仍是復原依據。

### 9.3 來源 DOM 改版

預期現象是 Action 失敗，舊 JSON 保留：

1. 不要刪除舊資料，也不要調低品質門檻只為了讓它變綠。
2. 保存失敗頁面的已去敏 fixture。
3. 修正 `services/parser.py` 與 parser fixture 測試。
4. 確認 `bangumi_id`、筆數、解析失敗率與圖片 URL。
5. 手動跑一季驗證後才恢復排程。

### 9.4 Cloudinary 圖片被誤刪

1. Disable 爬蟲與所有 retention。
2. 不要重新跑爬蟲覆蓋證據。
3. 使用 retention manifest、cache 與 Cloudinary 資產匯出確認 public IDs。
4. Cloudinary Assets → Deleted assets／Version History，從 backup restore。
5. 若沒有 backup，立即聯絡 Cloudinary Support；是否能救回取決於方案與刪除時間，不能保證。
6. Restore 後檢查 URL version。若 secure URL 版本改變，更新相應 JSON 與 cache，再通過全部 gate 後部署。
7. 抽查所有季度，不只檢查被清理的季度，因為相同內容圖片可能跨季度共用。

### 9.5 憑證再次外洩

1. 原平台立即 revoke／rotate。
2. 將 `CRAWLER_SCHEDULE_ENABLED=false` 並取消執行中的 job；必要時 Disable workflow。
3. 依外洩類型更新 `crawler-production`、`cloudinary-retention-plan`、`cloudinary-retention-execute`；若是 automation GitHub App private key，立即撤銷舊 key、產生新 key並更新 `automation-publisher` 的 `AUTOMATION_APP_PRIVATE_KEY`。
4. 檢查 GitHub audit／Actions logs、Cloudinary 使用紀錄、Discord 訊息與 Sentry ingestion。
5. 確認沒有異常上傳、刪除或通知。
6. 修正洩漏來源。
7. 最後評估 Git 歷史清理。

### 9.6 Cloudflare build 失敗

Cloudflare 會繼續提供上一個成功部署：

1. 開啟失敗 deployment log。
2. 檢查 Python 版本、`requirements-build.txt` hash、JSON schema 與 `manage.py validate-all`。
3. 不要在 Dashboard 暫時改成未鎖定的安裝命令。
4. 在本機以 `bash build.sh` 重現。
5. 修 Git 後重新部署。

## 10. Git 歷史清理：必須另開維護窗口

這不是一般部署的一部分，也不應由自動化執行。它會改變大量 commit SHA，可能讓舊 clone、fork、PR、簽章與 Cloudflare 連結失效。

只在下列條件都完成後安排：

- 所有外洩秘密已撤銷或輪替。
- 本次架構修補已合併且穩定。
- 所有協作者收到維護時間與重新 clone 指示。
- 已盤點 forks、open PRs、tags、releases 與部署整合。
- 有一份加密離線備份，且清楚知道其中仍含已失效秘密。
- 使用者明確同意 force-push。

### 10.1 維護窗口前

1. Disable GitHub workflows。
2. 暫停 Cloudflare automatic production builds。
3. 暫停所有人 push、merge 與 release。
4. 關閉或記錄 open PR；歷史重寫後舊 PR 可能無法正常合併。
5. 檢查 branch protection／ruleset，記錄原設定。
6. 安裝 `git-filter-repo` 2.47 以上與可信的秘密掃描器。

### 10.2 一律用全新 mirror clone

不要在日常工作目錄或目前這個 Codex 共享工作區執行。範例：

```powershell
Set-Location C:\safe-maintenance
git clone --mirror https://github.com/0961282056/anime-static-crawler.git anime-static-crawler-clean.git
Set-Location .\anime-static-crawler-clean.git
```

準備一個位於 repository 外的 replacement 檔，內容是已失效的舊秘密；不要把它加入 Git、雲端同步或命令列參數。先用測試 clone 確認規則。

移除所有歷史中的 `.env`，並同時替換可能被複製到其他檔案的舊值：

```powershell
git filter-repo `
  --sensitive-data-removal `
  --invert-paths `
  --path .env `
  --replace-text C:\secure\old-secrets-replacements.txt
```

若 `.env` 曾用其他路徑或名稱，必須為每個路徑加入額外 `--path`。不要把 `cloudinary_cache.json`、季度 JSON 或一般設定誤刪。

### 10.3 推送前驗證

- `git log --all -- .env` 應無結果。
- 使用歷史秘密掃描器掃描所有 refs，不只掃目前 branch。
- 檢查 `.git/filter-repo/changed-refs`，確認受影響 branch、tag 與 PR 數量符合預期。
- 抽查 main、tags 與資料檔仍存在。
- 在另一個目錄從清理後 mirror 建立工作 clone，執行 pytest、ruff、資料驗證與 build-only。
- 由第二人或維運紀錄再次確認，才進下一步。

### 10.4 force-push 是最後不可逆關卡

`git-filter-repo` 通常會移除 `origin` 以防誤推；確認後重新加入：

```powershell
git remote add origin https://github.com/0961282056/anime-static-crawler.git
git push --force --mirror origin
```

執行前可能需要暫時調整 ruleset。這會重寫全部 branch 與 tags；絕對不要在沒有維護窗口、驗證與明確同意時執行。

GitHub 的 `refs/pull/*` 是唯讀，force-push 無法清除所有 cached views。若舊秘密仍能透過 PR 或 cached commit 看見，聯絡 GitHub Support。Fork 必須由 fork 擁有者另外清理或刪除。

### 10.5 清理後

1. 所有協作者刪除舊 clone，重新 clone；不要對舊 clone 執行 pull 後再 push，否則秘密歷史可能回來。
2. 刪除舊本機 branch、worktree 與 CI cache。
3. 重新連接或驗證 Cloudflare Git integration，觸發一次 `main` production build。
4. 恢復 ruleset、branch protection、GitHub workflows 與 Cloudflare automatic builds。
5. 再跑一次 GitHub secret scanning／歷史掃描。
6. 聯絡 fork 擁有者並處理 GitHub cached PR refs。
7. 安全刪除 repository 外的 replacement 檔；密碼管理器只保留新秘密與輪替紀錄。
8. 觀察 GitHub、Cloudflare、Cloudinary、Discord 與 Sentry 24 小時。

歷史清理降低日後被看見的機率，但無法刪除別人的既有 clone，也不能取代最前面的 revoke／rotate。

## 11. 定期維運節奏

每月：

- 檢查 GitHub Action 成功率、資料筆數與 parse failures。
- 檢查 Cloudinary credits、asset 成長與 backup 狀態。
- 檢查 Sentry spike、允許網域與費用。
- 確認 Discord webhook 沒有陌生訊息。

每季：

- 更新依賴 lock 並跑完整 gate。
- 匯出 Cloudinary manifest。
- 演練 Cloudflare rollback 與一筆 Cloudinary restore。
- 檢查 GitHub Environment secrets 的擁有者與用途。
- 視需要產生 retention dry-run；不代表一定要刪。

每年或人員異動時：

- 輪替 Cloudinary key 與 Discord webhook。
- 檢查 GitHub、Cloudflare、Cloudinary、Sentry 成員權限與 MFA。
- 更新災難復原聯絡人、網域與平台設定紀錄。
