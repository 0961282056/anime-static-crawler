# 部署與平台設定手冊

這份手冊是給第一次接手專案的人。照順序完成即可，不需要先理解所有程式碼。

## 1. 你會用到的四個平台

| 平台 | 責任 | 是否保存秘密 |
|---|---|---|
| GitHub | 保存程式與資料、執行測試和每日爬蟲 | 是；Cloudinary、Discord 與 GitHub App private key |
| Cloudflare Pages | 將 `templates/`、`static/`、`dist/data/` 建置成網站 | 否；只負責 build-only |
| Cloudinary | 保存動畫封面圖片 | API secret 只交給爬蟲與人工 retention |
| Discord | 接收每日成功／失敗執行摘要 | Webhook URL 必須保密 |

正式資料流程只有一條：

1. GitHub Actions 從來源網站爬取。
2. 通過 schema 與品質 gate 後，原子寫入 `dist/data/*.json`。
3. GitHub 只有在資料或 `cloudinary_cache.json` 真正改變時才提交。
4. Cloudflare Pages 收到 `main` 的新提交後執行 `bash build.sh`。
5. `build.sh` 以 `requirements-build.txt` 安裝鎖定依賴，驗證資料，並產生 `dist/index.html`、`dist/static/`、`dist/_headers`。

## 2. 本機第一次安裝

### 2.1 安裝前準備

請先安裝：

- Git
- Python 3.11
- Windows：PowerShell；若要直接執行 `build.sh`，另需 Git Bash 或 WSL

確認版本：

```powershell
git --version
py -3.11 --version
```

專案的 `.python-version` 固定主版本為 3.11。不要使用 Python 3.10 產生新的 lock 或正式資料。

### 2.2 建立獨立環境

在專案根目錄執行：

```powershell
py -3.11 -m venv .venv
.\.venv\Scripts\Activate.ps1
python -m pip install --require-hashes -r requirements-dev.txt
```

若 PowerShell 不允許執行 `Activate.ps1`，不要為了這個專案降低全機安全設定；改用完整路徑：

```powershell
.\.venv\Scripts\python.exe -m pip install --require-hashes -r requirements-dev.txt
```

依賴檔用途：

| 檔案 | 用途 |
|---|---|
| `requirements-build.txt` | Cloudflare build-only 最小依賴 |
| `requirements.txt` | 正式爬蟲依賴 |
| `requirements-dev.txt` | 本機測試與品質工具，並包含開發所需依賴 |
| `*.in` | 人工維護的直接依賴來源；正式安裝以有 hash 的 `*.txt` 為準 |

不要手動編輯 hash lock 來「讓安裝通過」。依賴升級應另開變更、重新產生 lock 並跑完整測試。

### 2.3 執行品質檢查

```powershell
python -m pytest
python -m ruff check .
python -m ruff format --check .
python manage.py validate-data
```

任何一項失敗都先修正，不要部署。

### 2.4 本機 build-only

這個流程只讀現有 JSON，不會爬網站、不會上傳或刪除 Cloudinary 圖片：

```powershell
$env:BUILD_ONLY = "true"
python generate_static.py
python manage.py validate-all
Remove-Item Env:BUILD_ONLY
```

Linux、macOS 或 Git Bash：

```bash
BUILD_ONLY=true python generate_static.py
python manage.py validate-all
```

也可以直接模擬 Cloudflare：

```bash
bash build.sh
```

成功條件：

- `dist/index.html` 存在。
- `dist/static/` 與 `static/` 完全一致。
- `dist/_headers` 與根目錄 `_headers` 一致。
- 所有 `dist/data/*.json` 通過 schema 與品質驗證。

### 2.5 只有需要本機爬蟲時才建立 `.env`

```powershell
Copy-Item .env.example .env
```

用文字編輯器填入自己的值。`.env` 已被 `.gitignore` 忽略；仍應在提交前確認：

```powershell
git status --short
git check-ignore .env
```

輸出應顯示 `.env` 被忽略，`git status` 不應列出它。不要截圖、貼到 Issue、Discord 或工作流程 log。

執行正式本機爬蟲：

```powershell
python generate_static.py
```

這會更新最近一年到下一季、上傳必要圖片並建置網站，但不會執行 retention。第一次操作前請先完成 Cloudinary Backup 與本手冊後面的秘密設定。

### 2.6 一次性修復歷史 `未知ID`

`backfill_ids.py` 會重新讀取來源季度頁，以動畫名稱精確配對歷史資料。預設只 dry-run，不寫檔，也不需要 Cloudinary 憑證：

目前狀態（2026-07-13）：dry-run 與正式回填均已完成，36 季、共 2,067 筆資料已改用來源真實 ID；嚴格資料驗證通過，`未知ID` 為 0。請不要在另一台電腦、另一個工作目錄或另一條分支重複執行一次性回填。

```powershell
$env:CRAWLER_USER_AGENT = "anime-static-crawler/2.0 (+https://github.com/0961282056/anime-static-crawler)"
python backfill_ids.py
Remove-Item Env:CRAWLER_USER_AGENT
```

它會先檢查所有含 `未知ID` 的季度；任何季度有重複名稱、找不到精確名稱或來源錯誤，整個 dry-run 失敗且不寫入。不要猜 ID 或調低 gate。

只有 dry-run 全部成功、已建立修復前 tag，並人工檢查摘要後，才在獨立分支執行：

```powershell
$env:CRAWLER_USER_AGENT = "anime-static-crawler/2.0 (+https://github.com/0961282056/anime-static-crawler)"
python backfill_ids.py --execute --confirm BACKFILL_IDS
python manage.py validate-data
git diff -- dist/data
Remove-Item Env:CRAWLER_USER_AGENT
```

每個季度以原子方式寫入。這是一次性資料遷移，不要加入每日排程；變更應獨立提交，通過 CI 後再合併。

## 3. GitHub Environment 與 Secrets

### 3.1 為什麼使用 Environment

Environment 可以把正式憑證或自動發布權限限制在指定 job 與 `main`，避免一般測試或其他分支讀取。此專案使用四個名稱，大小寫與連字號必須一致：

- `crawler-production`：每日正式爬蟲。
- `cloudinary-retention-plan`：只產生 Cloudinary retention dry-run manifest。
- `cloudinary-retention-execute`：只供人工核准後執行已等待 30 天的 manifest。
- `automation-publisher`：只供不持有 Cloudinary 或 Discord 憑證的 publisher 建立 automation Pull Request 並要求 auto-merge。

### 3.2 建立 `crawler-production`

1. 開啟 GitHub 儲存庫。
2. 進入 **Settings → Environments → New environment**。
3. 名稱輸入 `crawler-production`。
4. Deployment branches 選擇只允許 `main`。
5. 每日排程需要自動執行，因此不要設定每次都要人工核准的 required reviewer；否則每天會卡在等待核准。
6. 在 **Environment variables** 與 **Environment secrets** 依下表新增項目。

| 名稱 | 類型 | 必要性 | 值從哪裡取得 |
|---|---|---|---|
| `CLOUDINARY_CLOUD_NAME` | Variable | 必要 | Cloudinary product environment；會出現在圖片公開網址 |
| `CRAWLER_USER_AGENT` | Variable | 必要 | `anime-static-crawler/2.0 (+https://github.com/0961282056/anime-static-crawler)` |
| `CLOUDINARY_API_KEY` | Secret | 必要 | Cloudinary Settings → API Keys |
| `CLOUDINARY_API_SECRET` | Secret | 必要 | 與新 API key 配對的 secret |
| `DISCORD_WEBHOOK_URL` | Secret | 正式 GitHub crawler 必要 | Discord 頻道 Webhook |

`GITHUB_TOKEN` 由 GitHub 每次執行自動建立，不需要自行新增。

`DISCORD_WEBHOOK_URL` 在執行 GitHub crawler（包含人工 Run workflow）前必須存在，並完成人工成功／失敗通知測試；本機執行仍可不設定 Discord。不要因為本機允許省略，就把正式 environment 的欄位留空。

另外在 **Settings → Secrets and variables → Actions → Variables** 建立 repository variable `CRAWLER_SCHEDULE_ENABLED`：

- 輪替與初次部署期間保持未設定或設為 `false`。
- 排程事件在不是 `true` 時會安全跳過，但 **Run workflow** 手動驗證仍可執行。
- 完成新 key 驗證與實站驗收後，才改為小寫字串 `true`。

排程時間是每天 UTC 17:00，也就是台灣時間 01:00；台灣沒有日光節約時間。

另有 **Daily Source Selector Canary** 在每天台灣時間 09:15 執行，也可人工 Run workflow。它只讀取當季來源 HTML，確認固定 selector 有抓到卡片、每張卡片都能通過 parser，且 `bangumi_id` 不重複；不建立 Anime crawler service、不下載圖片、不讀 Cloudinary secret，也不寫 JSON 或 cache。成功只寫入 GitHub step summary，不傳 Discord；失敗時才由獨立 job 使用 `crawler-production` 的 `DISCORD_WEBHOOK_URL` 傳送已去敏告警。

canary 合併到 `main` 後的第一次驗證：

1. 到 GitHub **Actions → Daily Source Selector Canary**。
2. 按 **Run workflow**，branch 選 `main`。
3. 等 `selector-canary` 綠燈；摘要應顯示季度、解析卡片數與來源 URL。
4. `notify-failure` 應顯示 skipped，Discord 不應收到成功訊息。
5. 如果 canary 失敗，只檢查 Actions 內的 parser／selector 錯誤；不要因此執行 retention，也不要手動覆寫正式 JSON。

GitHub secret 儲存後不能讀回。請把原始值另外保存在密碼管理器，不要保存在 README、筆記、試算表或聊天訊息。

若舊版曾把同名秘密放在 **Settings → Secrets and variables → Actions → Repository secrets**，先不要立刻刪除。先用 Environment secrets 手動執行新版 crawler 並驗證成功，然後刪除 Repository-level 的 `CLOUDINARY_API_KEY`、`CLOUDINARY_API_SECRET`、`DISCORD_WEBHOOK_URL` 等同名舊項目。最後再手動執行一次，確認工作流程只依賴 Environment；不要長期保留兩份同名秘密，以免日後更新錯位置。

### 3.3 建立 `cloudinary-retention-plan`

1. 以相同步驟建立 environment `cloudinary-retention-plan`。
2. 只允許 `main`。
3. 這個 environment 只供 dry-run；若有第二位維運者，仍建議設定 required reviewer。
4. 新增 Variable `CLOUDINARY_CLOUD_NAME`，以及 Secrets `CLOUDINARY_API_KEY`、`CLOUDINARY_API_SECRET`。

再建立 `cloudinary-retention-execute`：

1. Deployment branches 只允許 `main`。
2. 設定 required reviewer；這是唯一可以呼叫 Cloudinary delete API 的 environment，不可省略人工核准。
3. 取消勾選 **Allow administrators to bypass configured protection rules**。此選項只能在 GitHub Environment 畫面操作，公開 REST／GraphQL API 沒有更新欄位。
4. 單人維運時保持 **Prevent self-review** 關閉，否則啟動者無人可以核准；若日後有第二位維運者，應改為開啟。
5. 新增相同的 `CLOUDINARY_CLOUD_NAME` Variable、`CLOUDINARY_API_KEY` 與 `CLOUDINARY_API_SECRET` Secrets。
6. 不要放 Discord 或 GitHub write token；此 workflow 只需讀取 main、下載受驗證 artifact 與操作 Cloudinary。

設定後用無效 confirmation 做一次閘門測試：保持 `CRAWLER_SCHEDULE_ENABLED=true`，確認第一個 job 顯示 **Waiting for approval** 且沒有任何 step 啟動，然後直接取消，不要核准。不要為了這項測試把排程切成 `false`，也不要使用有效 manifest 或 `DELETE_UNREFERENCED`。

GitHub 不會在 environments 間自動複製 variables／secrets，三個 Cloudinary environments 都要個別更新。Cloudinary 每次輪替時也必須同步更新這三處；`automation-publisher` 不得放入任何 Cloudinary 憑證。

Cloudinary 若已啟用 2026 **Roles and Permissions**，新建 product-environment API key
預設沒有角色。請在 Cloudinary **Settings → API Keys** 為 key 指派目前 product
environment 的角色，否則憑證可能通過基本驗證但 Admin API 只回傳空清單或 `403
missing permissions (read)`。為相容此專案完整功能，可先用 product-environment
**Master Admin**；Enterprise 若能建立 granular custom roles，建議三個 GitHub
environments 使用三把不同 key，分別限制為 crawler 的讀取／上傳、plan 的唯讀，以及
execute 的讀取／刪除。輪替 dry-run 必須看到大於 0 的 `inventory`，不能把
`inventory=0` 當成成功。

### 3.4 建立 `automation-publisher` 與 GitHub App

自動資料更新與 retention cache 失效都必須走 Pull Request，不能直接推送 `main`。建立一個只服務此儲存庫的 GitHub App：

1. 在 GitHub 帳號或組織的 **Developer settings → GitHub Apps** 建立專用 App，並只安裝到 `0961282056/anime-static-crawler`。
2. Repository permissions 只給 **Contents: Read and write**、**Pull requests: Read and write**；Metadata 維持必要的唯讀預設。不要給 Administration、Actions write、Secrets 或 Environments 管理權。
3. 產生 private key，下載後立即放進密碼管理器；不要提交 `.pem`、貼到 Issue 或放進一般 repository secret。
4. 建立 environment `automation-publisher`，Deployment branches 只允許 `main`。每日 publisher 需要自動執行，不設定 required reviewer。
5. 在該 environment 建立 Variable `AUTOMATION_APP_CLIENT_ID` 與 Secret `AUTOMATION_APP_PRIVATE_KEY`。private key 必須完整保留 PEM 換行。
6. 此 environment 不得放 `CLOUDINARY_*`、`DISCORD_WEBHOOK_URL` 或其他正式服務憑證。
7. 在儲存庫 **Settings → General → Pull Requests** 同時啟用 **Allow squash merging** 與 **Allow auto-merge**。Publisher 只負責建立 `automation/...` 分支、開 Pull Request 並要求 squash auto-merge；是否真正合併仍由 ruleset 與 required check 決定。

使用專用 GitHub App token 是必要的。只用 workflow 內建 `GITHUB_TOKEN` 建立 PR，可能因 GitHub 的遞迴防護而不觸發 `pull_request` 品質流程。這個 App 與 GitHub Actions 都不可加入 ruleset bypass。

### 3.5 Actions 與分支保護

在 **Settings → Actions → General**：

- 預設 `GITHUB_TOKEN` 權限維持唯讀。
- crawler 與 retention 的秘密 job 保持 `contents: read`；只有掛載 `automation-publisher`、不持有正式服務秘密的 publisher job 可以取得 GitHub App token。
- 不要允許 fork PR 取得 secrets 或 write token。

第一次推送新版工作流程後，先讓它執行一次；再到 **Settings → Rules → Rulesets** 保護 `main`：

1. 建立只套用 `main` 的 branch ruleset。
2. 開啟 **Require a pull request before merging**；若每日更新要完全自動化，可把 required approvals 設為 0，但不得關閉 PR 要求。
3. 開啟 **Require status checks to pass**，加入 `Quality Gate / quality`（GitHub 畫面有時只顯示 `quality`），並要求 branch 在合併前保持最新。Pull Request 必須等測試、lint、資料驗證與 deterministic build 全綠。
4. 禁止直接 push、branch deletion 與 force-push。
5. **Bypass list 不加入 GitHub Actions，也不加入 automation GitHub App**；一般使用者與所有管理員同樣不應取得日常 bypass。
6. Git 歷史清理時若真的需要暫停規則，必須依安全手冊另開維護窗口，完成後立即恢復。

Crawler 先在 `crawl-and-prepare` 以唯讀 GitHub 權限完成 pip check、ruff、pytest、build-only、完整爬取與資料驗證，再上傳短期 artifact；後續 publisher 從 `automation-publisher` 取得短效 GitHub App token，建立只包含 `dist/data/` 與 `cloudinary_cache.json` 的 PR。Retention 採 cache-first 兩階段：受保護 job 先計算候選並產生 cache 失效 receipt，無 Cloudinary 秘密的 publisher 建立 cache-only PR；該 PR 通過 `Quality Gate / quality` 並合併後，required-reviewer job 才重新驗證 safe main 並刪除 Cloudinary 圖片。兩類 PR 都要求 branch 最新且不得直接推送 `main`。

## 4. Cloudflare Pages 設定

### 4.1 建立或修正專案

在 Cloudflare Dashboard：

1. 進入 **Workers & Pages**。
2. 選擇既有 Pages 專案；若沒有，選 **Create → Pages → Connect to Git**。
3. 連接 GitHub 儲存庫 `0961282056/anime-static-crawler`。
4. Production branch 設為 `main`。
5. Framework preset 選 `None`。
6. Root directory 留空或 `/`，代表儲存庫根目錄。
7. Build command 填 `bash build.sh`。
8. Build output directory 填 `dist`。
9. 在 Production 與 Preview 環境都新增 `SKIP_DEPENDENCY_INSTALL=1`，避免 Cloudflare 自動安裝依賴後又由 `build.sh` 重複安裝。
10. 儲存。

不要把 build command 改成直接執行未鎖依賴的 `pip install jinja2`；`build.sh` 才是唯一正式建置入口。

### 4.2 Cloudflare 環境變數

進入 Pages 專案的 **Settings → Environment variables**。

Production 建議值：

| 名稱 | 值 | 說明 |
|---|---|---|
| `SKIP_DEPENDENCY_INSTALL` | `1` | 關閉平台自動依賴安裝，統一交由 `bash build.sh` |

`.python-version` 已指定 Python 3.11。若 Cloudflare 實際 log 沒有採用，才另外設定 `PYTHON_VERSION=3.11`。

以下內容不得出現在 Cloudflare Pages：

- `CLOUDINARY_API_SECRET`
- Cloudinary API key pair
- `DISCORD_WEBHOOK_URL`
- 任何帳號密碼或 LINE token

Cloudflare 只做 build-only，不需要這些權限。

Preview 建議設定：

- `SKIP_DEPENDENCY_INSTALL=1`
- Preview branch controls 只允許明確需要的分支；不需要預覽時可關閉

Production 與 Preview 都要再次確認：Build command 是 `bash build.sh`，Build output directory 是 `dist`。兩個環境的變數分開管理，不要只設定 Production。

### 4.3 安全標頭

根目錄 `_headers` 是唯一來源；`generate_static.py` 會複製成 `dist/_headers`。目前包含：

- CSP
- `X-Content-Type-Options`
- `Referrer-Policy`
- 防 iframe 嵌入
- Permissions Policy
- HTML、JSON 與靜態資源快取規則

不要只在 Cloudflare Dashboard 手動改一份相同規則，否則日後無法知道哪裡才是正確來源。

目前 Alpine 使用標準 build，因此 CSP 暫時需要 `'unsafe-eval'`。若未來改成 `@alpinejs/csp`，必須先測完所有 `x-data`、`x-on` 與運算式，再移除；不能只刪 header 文字。

外部 CDN 資源必須：

- 鎖完整版本，不使用 `@11`、`latest` 等浮動網址。
- 同時提供正確 `integrity` 與 `crossorigin="anonymous"`。
- 上游檔案升級時重新計算 SRI，否則瀏覽器會拒絕載入。

## 5. Discord 設定

1. 在 Discord 伺服器開啟要接收通知的頻道。
2. 進入 **Edit Channel → Integrations → Webhooks**。
3. 建立專用 Webhook，例如 `Anime Crawler Production`。
4. 複製 URL 後立刻放入 GitHub `crawler-production` 的 `DISCORD_WEBHOOK_URL`。
5. 不要把 URL 貼進 `.env.example`、程式碼或 Issue。

Webhook URL 等同可發訊息的密碼。懷疑洩漏時應刪除舊 Webhook、建立新 Webhook，再更新 GitHub；只改名稱沒有用。

本機開發或一次性本機執行可以不設定 Discord；正式 GitHub 每日排程與人工 **Run workflow** 都把 `DISCORD_WEBHOOK_URL` 視為必要的唯一維運告警通道。未建立並驗證 webhook 前，`CRAWLER_SCHEDULE_ENABLED` 必須保持 `false`，且不要執行 GitHub crawler；本機 BUILD_ONLY／測試仍可進行。正式 workflow 中 Webhook 傳送失敗會讓 Action 失敗，錯誤只保留安全 HTTP 狀態碼，不會輸出 webhook URL。

selector canary 是例外：成功不通知，只有來源 selector／parser 契約失敗才發 Discord，避免每日重複訊息；通知不包含原始 exception 或來源 HTML。

Sentry 已停用。不要再新增 `SENTRY_DSN`、`SENTRY_BROWSER_DSN` 或 `APP_ENVIRONMENT`；舊的 GitHub secret、Cloudflare variable 與 Sentry project/client key 應撤銷或刪除。

## 6. 第一次正式部署

### 6.1 舊版升級時先完全停用舊 crawler

舊 workflow 尚未包含 `CRAWLER_SCHEDULE_ENABLED` 判斷，所以只把變數設成 `false` **不能**阻止舊排程。合併任何新版檔案前先做：

1. GitHub → **Actions** → 選擇舊的每日 crawler workflow。
2. 右上角選單按 **Disable workflow**。
3. 回到 workflow runs，逐一取消所有 Running 與 Pending 的 crawler jobs。
4. 確認舊 crawler 已不再直接寫入 `main` 或上傳 Cloudinary，也沒有 Running／Pending job。

新版 workflow 合併到 `main` 後，`CRAWLER_SCHEDULE_ENABLED` 才開始有效；在人工驗證完成前仍保持 `false`。

### 6.2 小白 Git 與 Pull Request 步驟

先把目前遠端 `main` 保存成 last-good tag。以下範例日期請換成實際部署日；tag 名稱不能重複：

```powershell
git switch main
git pull --ff-only
git tag -a last-good-pre-architecture-2026-07-13 -m "Last good before architecture upgrade"
git push origin last-good-pre-architecture-2026-07-13
git switch -c codex/architecture-hardening
```

完成檔案修改與本機檢查後：

```powershell
git status --short
git check-ignore .env
git add -A
git diff --cached --check
git status --short
git commit -m "refactor: harden crawler architecture"
git push -u origin codex/architecture-hardening
```

`git check-ignore .env` 應顯示 `.env`；若 `.env` 出現在 staged 清單，先停止，不要 commit。接著到 GitHub 儲存庫頁面：

1. 按 **Compare & pull request**，base 選 `main`、compare 選本次已推送的 `codex/...` 變更分支。
2. 建立 Pull Request，等待 required check `Quality Gate / quality` 成功。
3. 檢查 **Files changed** 沒有 `.env`、真實秘密、`pyc` 或手動生成的 `dist/static/`。
4. 綠燈後按 **Merge pull request**；不要用 force-push 或繞過品質檢查。
5. 合併後回本機執行 `git switch main`、`git pull --ff-only`，確認已取得新版 workflow。

若 tag 推送失敗並顯示名稱已存在，不要覆寫舊 tag；改用今天日期或加序號建立新的 last-good tag。

### 6.3 平台設定與上線順序

依序執行：

1. 保持舊 workflow Disabled，並將 repository variable `CRAWLER_SCHEDULE_ENABLED` 保持未設定或 `false`。
2. 完成安全手冊的 Cloudinary 新 key 輪替，但先保留舊 key 啟用。
3. 確認 `crawler-production` variables／secrets 已填好，包括正式排程必要的 `DISCORD_WEBHOOK_URL`。
4. 建立限本儲存庫使用的 automation GitHub App 與 `automation-publisher` environment，填入 `AUTOMATION_APP_CLIENT_ID`／`AUTOMATION_APP_PRIVATE_KEY`，並確認 App 與 GitHub Actions 都不在 ruleset bypass。
5. 確認 Cloudflare Production 與 Preview 都是 build=`bash build.sh`、output=`dist`、`SKIP_DEPENDENCY_INSTALL=1`。
6. 依上一節建立分支、通過 `Quality Gate / quality` 並合併到 `main`。
7. 到 Cloudflare **Deployments** 等待 production deployment 成功。
8. 先完成下一節的實站驗收。
9. 回 GitHub **Actions** 重新 **Enable workflow**，然後手動執行新版 crawler；安全開關為 `false` 仍允許手動執行。
10. 確認 cleanup／retention 沒有被執行。
11. 若資料有變，確認 bot 建立只含 `dist/data/` 與 `cloudinary_cache.json` 的 automation PR；等待 `Quality Gate / quality` 全綠、auto-merge 到 `main`，且來源 branch 被清理。若沒變，應顯示沒有 PR。
12. 等待 Cloudflare 因 automation PR 合併再次建置，再驗收一次。
13. Environment secrets 驗證成功後，刪除 Repository-level 的同名舊 secrets，再手動驗證一次。
14. 新 Cloudinary key 的爬取與上傳成功後，才停用舊 key。
15. 觀察 24 小時；確認穩定且 Discord 通知正常後，把 `CRAWLER_SCHEDULE_ENABLED` 設為 `true`，才恢復每日排程。

## 7. 部署驗收表

每次架構、依賴、模板或安全標頭有異動，都要完整勾選：

- [ ] GitHub 測試、lint、格式、資料 schema 與 build gate 全綠。
- [ ] `actionlint`、ShellCheck 與 Playwright smoke test 全綠。
- [ ] `main` 要求 Pull Request、branch 最新與 `Quality Gate / quality`，且 GitHub Actions／automation App 都不在 bypass list。
- [ ] `cloudinary-retention-execute` 有 Required reviewer、只允許 `main`，且 **Allow administrators to bypass configured protection rules** 已取消勾選。
- [ ] `automation-publisher` 只有 `AUTOMATION_APP_CLIENT_ID`／`AUTOMATION_APP_PRIVATE_KEY`，GitHub App 只安裝於本儲存庫且只具 Contents／Pull requests 讀寫權。
- [ ] Cloudflare production branch 是 `main`。
- [ ] Cloudflare Production 與 Preview 的 build command 都是 `bash build.sh`、output 都是 `dist`，且都有 `SKIP_DEPENDENCY_INSTALL=1`。
- [ ] 首頁能開啟，瀏覽器主控台沒有 JavaScript 或 CSP 錯誤。
- [ ] 當季資料可載入。
- [ ] 至少一個舊季度可載入，避免只測最新資料。
- [ ] 年份、季度、星期與關鍵字篩選正常。
- [ ] Cloudinary 圖片與無圖片 placeholder 正常。
- [ ] 加入分享清單、移除、產生分享圖正常。
- [ ] 手機寬度版面正常。
- [ ] `index.html` 回應包含 CSP、`nosniff` 與防 iframe 標頭。
- [ ] `data/*.json` 可讀且不是 error list、空資料或 HTML 錯誤頁。
- [ ] GitHub、Cloudflare、產出的 HTML 與 CSP 都沒有 Sentry 設定或 ingest 白名單。
- [ ] 正式排程使用的 Discord webhook 已設定，能收到合理的成功／失敗摘要，訊息沒有秘密。
- [ ] Daily Source Selector Canary 人工執行成功，`notify-failure` skipped，且沒有寫入資料或 Cloudinary。
- [ ] `CRAWLER_SCHEDULE_ENABLED` 只在輪替與人工驗證完成後設為 `true`。
- [ ] 有資料變動時只建立受 required check 保護的 automation PR；無資料變動時沒有 PR，沒有任何 workflow 直接推送 `main`。
- [ ] Cloudinary asset 數量沒有因一般爬蟲突然下降。

## 8. 平台設定異動摘要

若從舊版升級，控制台需要同步確認：

| 平台 | 必做異動 |
|---|---|
| GitHub | 升級前 Disable 舊 crawler 並取消 jobs；建立 `crawler-production`、`cloudinary-retention-plan`、`cloudinary-retention-execute`（execute 必須有 reviewer 且禁止 admin bypass）與 `automation-publisher`；安裝限本儲存庫、只具 Contents／Pull requests 讀寫權的 GitHub App；驗證後刪 Repository-level 同名 secrets；`main` 強制 PR、branch 最新與 `Quality Gate / quality`，GitHub Actions／App 都不得 bypass，automation PR 通過後才 auto-merge |
| Cloudflare | Production=`main`；Production／Preview 都設 build=`bash build.sh`、output=`dist`、`SKIP_DEPENDENCY_INSTALL=1`；刪除不必要的 Cloudinary／Discord secrets |
| Cloudinary | 產生專用新 access key、完成無停機輪替、啟用 automatic backup 與 existing assets 初始備份 |
| Discord | 正式排程必須使用專用 webhook；本機可省略；不再使用任何進入 Git 的舊 URL |

高風險操作（圖片刪除、Git force-push、舊密碼處理）請接著閱讀[安全維運與災難復原手冊](安全維運與災難復原.md)。
